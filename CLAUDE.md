# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Перед любой правкой кода — прочитай этот файл и применяй его правила. Если правила противоречат пользовательскому запросу — спроси, не действуй молча.**

## Команды

```bash
npm install          # один раз
npm run dev          # eleventy --serve --port=8765, live-reload, watch
npm run build        # сборка в _site/
npm run clean        # rm -rf _site/
```

Тестов нет — проверка ручная в браузере (см. раздел «Проверка работоспособности»).

## Карта архитектуры

Где что лежит — нужно знать перед правкой:

| Задача | Файл |
|---|---|
| Контент 4 страниц услуг (копи, тарифы, FAQ, hero-фразы) | `src/_data/services.js` — **единственный источник правды** |
| Бренд, контакты, домен, версии CSS/JS | `src/_data/site.js` |
| Скелет HTML (head/header/footer) | `src/_includes/layouts/base.njk` |
| Рендер страницы услуги из `services[serviceId]` | `src/_includes/layouts/service.njk` |
| Главная (FAQ + morph-фразы — хардкод, **не** из `services.js`) | `src/index.njk` |
| Партиалы (hero-terminal, pricing-cards, timeline, integrations, guarantees, final-cta, faq) | `src/_includes/partials/` |
| JSON-LD (ProfessionalService / Service / FAQPage / BreadcrumbList) | `src/_includes/partials/head-meta.njk` |
| Sitemap | `src/sitemap.njk` (генерится из `collections.all`) |
| Стили / JS | `src/static/styles.css`, `src/static/script.js` |
| Конфиг Eleventy | `.eleventy.js` |

Цепочка для страницы услуги: `src/<slug>.njk` (frontmatter с `serviceId`) → `layouts/service.njk` → `layouts/base.njk`.
Главная идёт в обход — `src/index.njk` напрямую `extends base.njk`.

## Подводные камни (coupling)

- **Порт 8765 продублирован** в `package.json` (`--port=8765`) и `.eleventy.js` (`setServerOptions`). Меняй в обоих местах.
- **Кеш-бастинг.** Правки в `static/styles.css` или `static/script.js` → инкрементировать `cssVersion` / `jsVersion` в `src/_data/site.js`. Иначе у части пользователей останется старая версия.
- **Домен — заглушка.** `https://example.com` в `_data/site.js → domain` пробрасывается во все canonical / OG / sitemap / JSON-LD. Менять в одном месте.
- **Defensive null-checks в `script.js`.** Pricing tabs и service preview существуют **только** на главной. Инициализаторы безопасно проходят по пустым `NodeList` на страницах услуг — **не убирай эти чеки**, иначе JS упадёт на `/sites/`, `/telegram-bots/` и т.п.
- **`prefers-reduced-motion`.** Lenis и плавный скролл отключаются, прелоадер ускоряется. Любая новая анимация должна это уважать.
- **Активная услуга в footer** помечена `aria-current="page"` (лаймовый цвет). Это не CSS-хак — атрибут проставляется в `partials/footer.njk` по `page.url`.

## 1. Перед тем как писать код

1. **Сначала читай [README.md](README.md).** Он отражает актуальное состояние проекта: структуру файлов, секции лендинга, поведение JS, контракты данных (`serviceContent`, `pricingContent`), SEO-заглушки, политику кеш-бастинга. Если README расходится с кодом — это баг README, скажи об этом.
2. **Найди в коде ровно ту часть, которой касается задача.** Открой и перечитай её. Не действуй по памяти.
3. **Составь и озвучь план.** До правок выпиши: какие файлы тронешь, какие — нет, и почему этого достаточно. План должен быть короче самой правки.
4. **Только после этого — патчи.** План → код, не наоборот.

## 2. Думай перед кодом

- **Явно формулируй допущения и план.** Прежде чем писать строки — скажи, что предполагаешь и что собираешься сделать.
- **Сомневаешься — спрашивай.** Не угадывай молча.
- **Если возможны несколько интерпретаций — перечисли их.** Пусть выбор сделает пользователь, не ты в одиночку.
- **Видишь более простой путь — скажи.** Возражай, если это оправдано.
- **Что-то непонятно — остановись.** Назови, что именно мешает, и задай вопрос. Лучше пауза, чем неправильная правка.

## 3. Хирургические правки

- **Не трогай код, которого проблема не касается.** Соседние функции, разметка рядом, чужие селекторы — мимо.
- **Не «улучшай» соседнее.** Никаких попутных рефакторингов, переименований, изменений форматирования и комментариев, если этого не просили.
- **Не переписывай работающее.** Если оно работает — оно остаётся.
- **Заметил не относящийся к задаче «мёртвый» код — упомяни в ответе, но не удаляй.** Удаление — отдельная задача с отдельным согласованием.
- **Не правь файлы вне задачи** (например, `sitemap.xml`, `robots.txt`, `og-image.png`), если правка их не требует напрямую.

## 4. Объём решения соразмерен задаче

- **Если задача решается в несколько строк без потери качества — оставь её в нескольких строках.** Не раздувай ради «архитектурности».
- **Не вводи абстракции под гипотетическое будущее.** Три похожие строки лучше преждевременной обёртки.
- **Не добавляй обработку ошибок и валидацию для невозможных сценариев.** Только на реальных границах (вход пользователя, внешний API).
- **Никаких полу-готовых заглушек.** Либо доделываешь, либо не начинаешь.

## 5. Стандарт инженерной работы (из AGENTS.md)

- Действуй как сениор: найди настоящую проблему, выбери минимальное корректное решение, сохраняй существующую архитектуру.
- Сохраняй кодировку UTF-8 (русский контент). Не используй инструменты, которые ломают кодировку.
- Никаких массовых автоматических переписываний — только точечные правки.
- Не откатывай чужие/локальные изменения пользователя.

## 6. Проверка работоспособности

**Каждое изменение покрывай реальной проверкой**, в том числе в разных ситуациях:

- **Синтаксис.** HTML/CSS/JS — без ошибок в консоли.
- **Браузер.** Открой страницу, прокликай изменённое.
- **Десктоп и мобильный.** Проверь обе ширины (особенно меню, hero, цены, FAQ).
- **Состояния.** Hover/focus/active, открытое мобильное меню, переключение табов с клавиатуры (`ArrowLeft/Right`, `Home`, `End`, `Enter`, `Space`), `Escape` для закрытия меню.
- **`prefers-reduced-motion`.** Анимации должны вести себя адекватно (прелоадер ускоряется, Lenis отключается, морфинг — мягко).
- **Регрессии.** Убедись, что не сломались соседние секции, скролл по якорям, прелоадер, кастомный курсор.
- **Если UI/тесты невозможно проверить** — скажи об этом прямо, не делай вид, что всё ок.

## 7. Кеш-бастинг

При значимых правках `styles.css` или `script.js` — обнови суффикс `?v=...` у соответствующего тега в [index.html](index.html). Иначе у части пользователей останется старая версия.

## 8. Коммиты

- **После каждого осмысленного изменения — отдельный коммит.** Не накапливай разнородные правки в одном.
- Сообщения коммитов короткие, по делу, в стиле уже существующих в репозитории (`git log` подскажет).
- Не делай `--amend`, не делай `--no-verify`, не делай force-push без явной просьбы.

## 9. Когда не уверен

Спроси пользователя. Стоимость уточнения низкая, стоимость неправильной правки — высокая.

## 10. Ведение MEMORY.md

[MEMORY.md](MEMORY.md) — журнал инженерных решений и подводных камней по проекту между сессиями. Цель: будущий Claude (и человек) не наступает на одни и те же грабли.

**Что писать:**

- **Решения с обоснованием:** что выбрали и **почему**, какие альтернативы отбросили.
- **Подводные камни в окружении / тулинге:** несовместимости версий, особенности Windows-путей, конфликты портов, поведение npx/npm.
- **Гипотезы и их проверка:** что проверили в браузере / на устройстве, что подтвердилось, что нет.

**Что НЕ писать:**

- Состояние задачи «в процессе» — это в чате, не в MEMORY.
- Дублирование README.md / CLAUDE.md.
- Личные TODO без срока (для TODO есть секция «Deferred / TODO» в README).

**Когда обновлять:**

- Сразу после решения, неочевидного из кода или README.
- Сразу после открытия подводного камня (особенно тулинг и окружение).
- В той же сессии, что и сама правка. Откладывание = забывание.

**Формат:**

- Обратный хронологический порядок: новое — сверху.
- Каждая запись: `## YYYY-MM-DD — короткий заголовок`, ниже 1–5 строк сути.
- Без воды и формализма. Пиши так, чтобы через полгода ты сам понял, что имел в виду.
